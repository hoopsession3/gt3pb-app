# Accessibility + AI/prefill sweep — 2026-07-17

Triggered by two things in the same conversation: a screenshot of Shoot Planner asking why AI wasn't
already reading dropped photos and building the shot list, and a screenshot of Studio's grid next to a
Canva mockup asking why it didn't look as clean. That turned into an architecture reconciliation (see
`lib/architecture.ts`'s Jul 16 pass) plus a "go wide" request: audit the whole app for accessibility
gaps and places AI/prefill should exist but doesn't.

Method: four parallel passes — customer-facing surfaces, crew console ops/business lanes,
Studio/content/brand (Shoot Planner prioritized), and a dedicated accessibility sweep (contrast,
headings, keyboard, forms, ARIA) — each reading its cluster in full rather than sampling. This doc
triages everything they found. **P1 is fixed and shipped this round.** P2 is real, scoped precisely
enough to build next round, but was more than one round could safely carry alongside P1. P3 is
deferred on purpose — either a genuine product/design call or too large to safely batch.

The two standing P0s from the original front-end audit — booking requests having no customer-facing
view (RLS-blocked), and office invoices having no in-app payment path — are untouched this round and
still need your call; see `FRONTEND_AUDIT_2026-07-16.md` for the full writeup, not repeated here.

## P1 — fixed this round

- **`kit.tsx`'s `SectionHeader` rendered no real heading element** — just a `<span>` styled to look
  like one. This one component backs every section title app-wide (83 call sites across 37 files per
  an exhaustive count), so it was the single highest-leverage gap found: a screen-reader user had
  effectively no heading outline on any screen built from the kit. Changed the label span to a real
  `<h2>` (correct under `AppShell`'s one-`<h1>`-per-route contract — every existing `SectionHeader`
  call is a subsection, never the page title). Zero visual change: `.k-sec .l` already set its own
  font/size/weight/color; added `margin:0` defensively since h2's browser-default margin wasn't
  previously being overridden by anything that mattered (span has no default margin).
- **Duplicate `<h1>` collisions, the same bug class as the earlier `/events` fix** — three distinct
  sources, three distinct fixes rather than one blanket rule:
  - `SignIn.tsx` rendered its own `<h1>` in all four of its steps. `SignIn` is embedded (not a
    dedicated route) in at least 8 places — `app/office`, `app/academy`, `app/crew`, `app/3mpire`,
    `app/driver`, `Reserves.tsx`, `OrderFunnel.tsx`, `RsvpRow.tsx` — most of which are *not* in
    `AppShell`'s `H1_SKIP`, so `AppShell`'s sr-only `<h1>` and SignIn's own `<h1>` both existed
    whenever an unauthenticated visitor hit any of them. Rather than auditing and registering each of
    those 8 call sites (and re-auditing every future one), fixed it at the source: `SignIn`'s heading
    is now `<h2>`, since it never owns a route outright. This fixes all 8 current sites and every
    future one for free, and is exactly correct semantically (a real `<h2>` under whatever ambient
    `<h1>` the embedding page or `AppShell`'s fallback provides).
  - `MarketingSplash.tsx` rendered a real `<h1>` for its promotional headline — but it's an overlay
    that mounts *on top of* whatever route is already showing, not a replacement for it, so it was a
    second `<h1>` on every customer-facing route for as long as the splash was up (its scrim already
    carries its own `aria-label`, so the heading wasn't even load-bearing for a11y). Changed to `<p>`;
    `.spl-head`'s CSS already set explicit font-size/weight/margin, so this is a zero-visual-change fix.
  - `app/academy/page.tsx` renders its own `<h1>` in two of its branches (disabled-backend, and the
    main signed-in view) — added `/academy` to `H1_SKIP`. That would have left the "you're not staff"
    wall branch with *no* heading at all (it used a `<div className="h-title">`, not an `<h1>`) —
    promoted that div to a real `<h1>`, consistent with the same class already being used as an `<h1>`
    two branches down in the same file.
- **Five keyboard-dead `role="button"` elements** — hand-rolled `role="button" tabIndex={0}` with no
  `onKeyDown`, so a keyboard user could tab to them but never activate them: `BrandCalendar.tsx:227`
  and `CompanyCalendar.tsx:352,373` (calendar day cells), `Studio.tsx:675,676` (the remove/make-cover
  controls on a media thumbnail). Swapped all five onto `lib/a11y.ts`'s existing `clickable()` helper
  — the two calendar files had no other event handlers competing with the new one, so it's a straight
  spread; the two Studio spans keep their existing `onClick` (which needs `stopPropagation` — the
  thumbnail itself is a real `<button>`, so the click can't bubble into it) and gained `clickable()`'s
  role/tabIndex/onKeyDown alongside it. Checked a sixth candidate, `PrepBoard.tsx:179`, and it turned
  out to already be correctly hand-rolled (has its own matching `onKeyDown`) — left alone.
- **Wired the AI shot-list drafting agent Ryan asked about directly.** Shoot Planner had a shot list
  you built by hand, one `+ Shot` at a time, with nothing reading the shoot's own context. Added
  `app/api/agents/shotlist` — same "propose it, the crew approves what to keep" shape as the day
  planner's `draft_day` and Studio's `studio-photo`: given the shoot's title/location and a few
  freeform notes, Sonnet (open-ended planning tier, not Haiku) proposes a concrete, ordered shot list;
  nothing writes to the database until the crew ticks which ones to keep. `ShootPlanner.tsx` gained a
  "✨ Draft shots with AI" trigger inside each open shoot (`ShotDraftPanel`, mirroring
  `EventDayPlanner.tsx`'s `DraftPanel` almost exactly) and the same claim-safety guard every other
  agent in the app carries. Deliberately scoped *without* the event/stop → shoot link (see P2) — that
  needs a migration and a UI decision about auto-creating shoots from events, this doesn't.
- **`window.prompt()`, three places** — `VipQueue.tsx` (verify/reject) and `PipelinePanel.tsx`
  (marking an opportunity "lost") were the only spots in the app still using a native browser prompt:
  unstyled, blocks the whole page, no tap-outside-to-cancel on mobile the way every other input here
  has. Built `components/PromptSheet.tsx` — a single-field input in the app's canonical `Sheet` — and
  rewired all three call sites onto it. Same cancel semantics as before (canceling the sheet aborts
  the action entirely, same as `window.prompt()` returning `null`).

Build: `npm run build` clean, 0 TypeScript errors, 80 routes (79 + the new `/api/agents/shotlist`).

## P2 — real, scoped, recommended next round

- **Shoot Planner's other headline gap: no link from an event/stop to a shoot.** Planning a shoot
  today means retyping the date/location/call-time that already live on the event or stop — the same
  disconnect Ryan's original screenshot was reacting to, half-solved by the AI drafting agent above.
  The `shoots` table (`0214_shoots.sql`) has no `event_id`/`stop_id` column, so this needs a
  **migration** (additive: nullable FK column + index) plus a UI decision I didn't want to guess at —
  should opening "Plan a shoot" from an event auto-create a shoot pre-filled from it, or just offer to
  link an existing one? Either is a clean build once you pick.
- **Three `window.prompt()`-adjacent patterns already fixed** surfaced one more worth a look:
  `CompanyCalendar.tsx` and `BrandCalendar.tsx` still use `window.confirm()` for delete confirmations
  (`ShootPlanner.tsx`'s `delShoot` does too). Native `confirm()` shares the same "unstyled, blocks the
  page" complaint as `prompt()` — didn't fold it into this round since a confirm-style `Sheet` variant
  is a slightly different shape than `PromptSheet`, but it's the same fix pattern.
- **CommandPalette's combobox/listbox ARIA wiring** — flagged by the accessibility pass as
  mechanically well-defined (the roles/attributes needed are a known checklist) but needs careful
  interaction testing against real keyboard nav before shipping, not a blind attribute add.
- **The `--red` token (#B82420) fails WCAG AA as a text color in the default dark theme** (~35 usages)
  — `--red-h` (#d64b46) already exists in the palette and passes. The existing contrast audit in
  `globals.css` (the "Day-mode legibility floor," dated Jul 9) is explicitly scoped to the crew-day
  light theme only; the default theme's tokens were carried forward, not re-verified, during an
  earlier "Pass-2 token extraction." Didn't touch this round — swapping a color token used 35 places
  needs a visual pass to confirm nothing was relying on the specific shade, not just a contrast ratio.
- **Agent 2's receipt → expense entry has no prefill** — logging an expense from a photographed
  receipt currently means retyping every field by hand even though Smart Intake's read-a-photo pattern
  already exists and is proven (this is now the fourth place it would apply: assets/inventory/
  documents, Studio content, and Shoot Planner's context, this being the fourth). Didn't build this
  round — wanted to confirm the exact expense-entry surface and field set against the live UI rather
  than the survey pass's summary before committing to an agent contract.
- **`ShootPlanner.tsx`'s own "+ Shot" / "+ New shoot" buttons render with no matching CSS** — they
  pass `className="shoot-add"` / `className="shoot-new"` to `InlineCreate`, overriding its default
  `adm-btn` styling, but neither `.shoot-add` nor `.shoot-new` has a rule in `globals.css`. Noticed
  while wiring the AI draft trigger next to them; low effort (add one rule, probably matching
  `.dp-add`'s), left alone this round to keep this round's diff to what was actually asked for.

## P3 — deferred: product/design calls, or feature-sized

- **109 unlabeled form fields, 14 unlabeled close buttons** (accessibility pass's counts) — real, but
  an app-wide sweep at this scale deserves its own dedicated round with its own build-and-verify pass,
  not a rider on this one. Worth prioritizing by traffic (customer checkout/booking fields first) when
  it's scheduled.
- **Whether to normalize 13 `qd-x` close buttons that carry a `title` attribute instead of
  `aria-label`** — functionally fine in most screen readers (title falls back), but inconsistent with
  the `aria-label` pattern used elsewhere. A tone/consistency call, not a bug.
- **Studio's focal-point picker has no keyboard interaction path** — needs an actual interaction
  design (arrow keys to nudge? a numeric fallback?), not a mechanical fix.
- **`GenerateDay.tsx`'s "Dialed For Today" stack recommender has no AI assist**, unlike its siblings.
  Flagged as lower-confidence: it's a currently-working deterministic feature, and swapping in an AI
  call trades a free, instant, always-available answer for one with cost/latency/claim-safety
  tradeoffs — worth deciding deliberately, not defaulting into.
- **A handful of "possibly intentional" gaps** the survey agents flagged without full conviction:
  BrandKit/LetterFlyer/SiteCopyEditor/ProposalDesk each had smaller findings (styling/prefill
  opportunities) that didn't get independently re-verified against current code this round the way
  the P1 items above did — listed here for visibility, not asserted as confirmed bugs. Worth a
  targeted look before building against them.

## What's already solid, for contrast

The `clickable()` helper itself (`lib/a11y.ts`) is correctly used everywhere it currently appears
(`app/3mpire`, `app/menu`, `StampCard.tsx`) — the five fixes above were the only drift from it, not a
sign the pattern itself is unreliable. `Sheet.tsx`'s focus trap, Escape-to-close, and focus
save/restore are solid and are exactly why `PromptSheet` and `ShotDraftPanel` could be built as thin
wrappers around it rather than reimplementing modal behavior. The claim-safety guard
(`lib/claimGuard.ts`) ran clean on every test draft during `shotlist` development — no changes needed
there.
